# Data Architecture

The application implements a resilient, multi-tiered data architecture designed to guarantee 100% UI uptime.

## The Tri-Layer Fallback
When a component requests data (e.g., `getCharacters()`), the architecture executes the following cascade:

1. **Live APIs (Top Priority):**
   - `thronesApi.js` and `iceAndFireApi.js` reach out to external servers. If successful, this data is used to populate volatile fields (like dynamic image URLs or exact titles).
2. **Local JSON (Secondary Priority):**
   - Derived datasets located in `public/data/` (e.g., `characters.json`) generated by Python pipelines. These provide complex metadata (e.g., spoken line counts, survival status) not available in the public APIs.
3. **Static Content (Ultimate Fallback):**
   - Found in `src/content/`. If the APIs fail or rate-limit the client, the UI falls back entirely to these static definitions. They contain the base structure (Biographies, Quotes, base relationships).

## Normalization
The `mergeService.js` is the heart of this architecture. It accepts data from all three layers and merges them using Optional Chaining (`?.`) and Nullish Coalescing (`??`) to output a single, predictable, flat object.

Example:
```javascript
return {
  ...baseContent,
  name: apiChar?.fullName || baseContent.name,
  image: apiChar?.imageUrl || null,
  popularity: localData?.popularity || null,
}
```
This guarantees the React components never crash due to a missing API field.
