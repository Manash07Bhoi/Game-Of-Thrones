# Project Memory Bank

*Created and Developed by Roshan.*

This document serves as the absolute source of truth for the architectural philosophy, technical constraints, and structural design of the Game of Thrones Westeros Portal. A new developer should be able to read this file and understand exactly *why* the project is built the way it is.

## Project Vision
**Goals:** To construct a structurally sound, highly performant, serverless/static portal for exploring the lore of Game of Thrones that feels cinematic, premium, and deeply immersive.
**Design Philosophy:** The UI must feel heavy, historic, and cinematic. We achieve this by avoiding heavy CSS frameworks (like Bootstrap or MUI) in favor of raw CSS, relying on pure CSS Grid/Flexbox, GSAP for timeline scrubbing, and IntersectionObserver for high-performance scroll reveals.

## Architecture Decisions

### Routing & GitHub Pages
*   **Decision:** Use `react-router-dom` with `BrowserRouter`. Avoid `HashRouter` to maintain clean, professional URLs.
*   **The GitHub Pages Constraint:** GitHub Pages does not support SPA HTML5 pushState. A direct visit to `/characters/jon-snow` results in a physical 404.
*   **The Solution:** We implemented the "SPA redirect hack". `public/404.html` catches the 404 and uses vanilla JS to rewrite the URL as a query parameter (e.g., `/?/characters/jon-snow`). `src/main.jsx` intercepts this parameter before React boots up and silently replaces the history state. This creates the illusion of native server-side routing on a static host.

### The Service Layer
*   **Decision:** React components must **never** perform direct network requests (`fetch`).
*   **Rationale:** Tight coupling of UI and Data fetching makes refactoring impossible. By forcing all components to call `contentService.js`, we established a universal abstraction layer. If we switch from ThronesAPI to a proprietary PostgreSQL database tomorrow, the React components remain entirely untouched.

### Cache Decisions
*   **Decision:** Implement a dual-tier cache (`cacheService.js`) using an in-memory Map and `localStorage` with a 24-hour Time-To-Live (TTL).
*   **Rationale:** The Ice and Fire API is rate-limited and heavily paginated. If the user hard-refreshes the page, `localStorage` prevents a redundant network call. If the user navigates between routes within the SPA, the in-memory Map serves the data in ~0ms, avoiding even the parsing overhead of `localStorage`.

### Data Merge Decisions
*   **Decision:** Multi-tiered data merging (`mergeService.js`).
*   **Rationale:** APIs fail. To guarantee 100% UI uptime, data is assembled in a cascading fallback priority:
    1.  **Live APIs:** (ThronesAPI, An API of Ice and Fire) provide dynamic images and live metadata.
    2.  **Local Datasets:** (`public/data/*.json`) provide derived metadata (e.g., spoken line counts) generated by offline Python pipelines.
    3.  **Static Content:** (`src/content/`) acts as the ultimate bedrock fallback, providing robust biographies and base structures if everything else fails.

## Known APIs
*   **ThronesAPI (`thronesApi.js`):** Fast, reliable, provides excellent image URLs. Primary source for Character data.
*   **Ice and Fire API (`iceAndFireApi.js`):** Slower, extremely dense with metadata, heavily paginated. Primary source for House data. Handled carefully in the service layer to avoid hanging the UI.

## Deployment Strategy
*   **Platform:** GitHub Pages via GitHub Actions.
*   **Build Pipeline:** `vite.config.js` sets the base path to match the repository name exactly (`/Game-Of-Thrones/`). The `.github/workflows/deploy.yml` action runs `npm ci` and `npm run build`, uploading the `dist/` directory as a Pages artifact.

## Repository Standards
*   **Asset Preservation:** **Never delete original data files or ZIP archives** located in `database/`. These represent the original provenance of the dataset and are required for the Python pipelines (`database/pipeline.py`) to generate the `public/data/` JSONs.
*   **Developer Credit:** "Created and Developed by Roshan" must be maintained in the documentation.

## Future Expansion Plans
*   **Search Engine:** Client-side integration using Fuse.js targeting the `search-index.json`.
*   **Interactive Maps:** Pan/zoom map of Westeros plotting the `locations.json` data.
*   **Family Trees:** D3.js or custom SVG generation mapping the lineages of the Great Houses.
*   **Timelines:** A dedicated horizontal scrolling component detailing the entire history of Westeros.

## Lessons Learned
*   Mobile overlay z-indexes are tricky when mixed with scroll-reveal transformations. Fixed by explicitly defining `z-index: 60` on the hamburger button and strictly locking body scroll via `document.body.style.overflow = 'hidden'`.
